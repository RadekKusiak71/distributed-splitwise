# Splitwise — Microservices Expense Sharing Platform

A microservices-based expense-sharing application that calculates optimized settlements from CSV files. Users upload expense data, and the system asynchronously computes minimal transfers needed to settle all debts.

## Services

| Service | Language | Framework | Port | Purpose |
|---------|----------|-----------|------|---------|
| **API Gateway** | Go 1.25 | Chi v5 | 8080 | JWT validation, request routing |
| **Identity Service** | Python 3.13 | FastAPI | internal | User registration & authentication |
| **Requests Service** | Go 1.25 | Chi v5 | internal | File uploads, request management |
| **Splitwise Worker** | Python 3.13 | — | — | Async CSV processing & settlement calculation |

## Request Flow

```
1. Register   POST /api/v1/users/register  →  Identity Service
2. Login      POST /api/v1/auth/login       →  Identity Service  →  JWT token
3. Upload     POST /api/v1/requests         →  Requests Service  →  S3 + SQS
4. Process    Worker polls SQS              →  Download CSV → Calculate → Upload result
5. Fetch      GET  /api/v1/requests/{id}    →  Requests Service  →  S3 links
```

## CSV Format

### Input (uploaded by user)
```csv
spender,debetor,value
Alice,Bob,100
Bob,Charlie,50
Charlie,Alice,30
```

### Output (generated by worker)
The worker calculates optimized settlements — minimal transfers to settle all debts:
```csv
debetor,spender,value
Bob,Alice,20.00
Charlie,Bob,50.00
```

### Settlement Algorithm
1. Calculate net balance per person: `amount_spent - amount_owed`
2. Separate into debtors (negative balance) and creditors (positive balance)
3. Greedy matching: pair the smallest debtor with the largest creditor, settle the minimum of their balances, repeat until all settled

## Getting Started

### Prerequisites
- Docker & Docker Compose
- AWS account with S3 bucket and SQS queue (or LocalStack for local development)

### Environment Variables

Each service requires a `.env` file in its directory.

**gateway-service/.env**
```env
JWT_SECRET=your_jwt_secret
IDENTITY_SERVICE_URL=http://identity-service:80
REQUESTS_SERVICE_URL=http://requests-service:80
```

**identity-service/.env**
```env
IS_DEBUG=true
DB_DSN=postgresql+psycopg://postgres:postgres@identity-service-db:5432/identity_db
JWT_SECRET_KEY=your_jwt_secret
JWT_ACCESS_TOKEN_TTL=1440
```

**requests-service/.env**
```env
API_PORT=80
DB_HOST=requests-service-db
DB_PORT=5432
DB_USER=postgres
DB_PASSWORD=postgres
DB_NAME=requests_db
JWT_SECRET_KEY=your_jwt_secret
AWS_ACCESS_KEY=your_access_key
AWS_SECRET_KEY=your_secret_key
AWS_REGION=your_region
AWS_BUCKET_NAME=your_bucket
AWS_S3_BASE_URL=https://your_bucket.s3.your_region.amazonaws.com/
AWS_SQS_QUEUE_URL=https://sqs.your_region.amazonaws.com/your_account/your_queue
```

**splitwise-worker/.env**
```env
POSTGRES_DNS=host=requests-service-db port=5432 user=postgres password=postgres dbname=requests_db
AWS_ACCESS_KEY=your_access_key
AWS_SECRET_KEY=your_secret_key
AWS_S3_REGION=your_region
AWS_S3_BUCKET_NAME=your_bucket
AWS_QUEUE_URL=https://sqs.your_region.amazonaws.com/your_account/your_queue
```

> **Important:** `JWT_SECRET` / `JWT_SECRET_KEY` must be the same across gateway, identity, and requests services.

```bash

```

**Identity Service** uses Alembic:

## Service Details

### API Gateway (Go)

Reverse proxy that routes traffic to downstream services and enforces JWT authentication on protected routes.

- **Public routes** (no auth): `/api/v1/users/*`, `/api/v1/auth/*`
- **Protected routes** (JWT required): `/api/v1/requests*`
- On protected routes, the gateway validates the JWT, extracts `user_id` from claims, and forwards it as the `X-User-ID` header

**Tech:** Go 1.25, Chi v5, golang-jwt

### Identity Service (Python)

Handles user registration and authentication. Issues JWT tokens on successful login.

- Passwords hashed with bcrypt
- JWT tokens use HS256 algorithm with configurable TTL (default 24h)
- Async PostgreSQL via SQLAlchemy 2.0 + psycopg

**Tech:** Python 3.13, FastAPI, SQLAlchemy, Alembic, bcrypt, PyJWT

### Requests Service (Go)

Manages file upload requests. Accepts CSV files, uploads them to S3, creates a database record, and publishes a processing job to SQS.

- Only `.csv` files are accepted
- Idempotency enforced via `X-Idempotency-Key` header (UUID)
- Request statuses: `pending` → `processing` → `completed` / `failed`
- Authorization: users can only access their own requests

**Tech:** Go 1.25, Chi v5, pgx, AWS SDK v2 (S3 + SQS)

### Splitwise Worker (Python)

Background worker that continuously polls SQS for processing jobs. Downloads CSV from S3, calculates optimal expense settlements, uploads the result CSV, and updates the request status in the database.

- Long-polling SQS with 20-second wait time
- Failed messages stay in queue for retry
- JSON structured logging

**Tech:** Python 3.13, boto3, pandas, psycopg

## Database Schemas

### Identity DB — `users`
| Column | Type | Constraints |
|--------|------|-------------|
| id | UUID | Primary Key |
| email | VARCHAR | Unique, Indexed |
| password | VARCHAR | bcrypt hash |
| created_at | TIMESTAMPTZ | Auto-generated |

### Requests DB — `requests`
| Column | Type | Constraints |
|--------|------|-------------|
| id | UUID | Primary Key, Auto-generated |
| user_id | VARCHAR(255) | Not Null |
| idempotency_key | VARCHAR(255) | Not Null |
| status | VARCHAR(50) | Default: 'pending' |
| input_s3_key | TEXT | Not Null |
| output_s3_key | TEXT | Nullable |
| created_at | TIMESTAMPTZ | Auto-generated |
| updated_at | TIMESTAMPTZ | Auto-generated |

Index: `idx_requests_status_created_at` on `(status, created_at)`

## Network Architecture

```
gateway-net (bridge)        ← API Gateway, Identity Service, Requests Service
identity-db-net (internal)  ← Identity Service ↔ Identity DB
requests-db-net (internal)  ← Requests Service ↔ Requests DB ↔ Splitwise Worker
backend-net (bridge)        ← Identity Service, Requests Service, Splitwise Worker
```

Only the API Gateway exposes a port to the host (8080). Database networks are internal — not reachable from outside Docker.
